1 (define symbol value)

Sets the current value of the symbol. symbol is not evaluated, value is evaluated.

Examples -

This example simply evaluates value and assigns it as the value of the given symbol.

1 (define symbol value)

The second example provides additional documentation about the symbol being defined, using an arbitrary property structure identified by the keyword doc at the head of the documentation block, followed by an association list of the documentation writer's own choosing. Typical documentation property words include usage for a brief description of the usage syntax, descr for a more elaborate description of the object being created, and brief for a brief textual description of the purpose of the object.

1 (define symbol
2  (doc (usage "(symbol param1 param2)")
3  (descr "Computes symbol")
4  (param1 "The first value")
5  (param2 "The second value"))
6  (fn (p1 p2) (+ p1 p2)))

The defined function takes two arguments and returns their sum. Pretty lame function, but serves to illustrate.

If you want the documentation block to be ignored by the reader, then you should call muse_init_env with the MUSE_DISCARD_DOC parameter set to MUSE_TRUE.

Generic functions
If the given symbol is already bound to a generic function, define behaves like define-extension - i.e. it extends the definition of the generic function with the new function's case.
Declarations
If you're defining a function and you leave its body empty - for example -
1 (define (f x y))
or
1 (define f (fn (x y)))
, it is taken to mean a declaration of the function's argument pattern. The purpose of a declaration facility is to permit mutually recursive functions in local scopes. The body of the declared function may be provided later on by redefinition using a similar argument pattern. For example, after the declaration shown in example above, you can provide a definition like -
1 (define (f x y) (+ (* x x) (* y y)))
or
1 (define f (fn (x y) (+ (* x x) (* y y))))
At the definition point, you can use any argument pattern that will match against the declared pattern. So you could have done -
1 (define (f a b) (+ (* a a) (* b b)))
which will be accepted. However, if you provide an incompatible argument pattern such as
1 (define (f x) (* x x))
, an error message will be shown. If you ignore the error message, the new argument pattern is the one that will take effect. Pattern matching is therefore useful to declare that something is a function without dscribing any of its arguments like this -
1 (define f (fn _))
. All argument patterns will match against the _ pattern (or any symbol in its place).

1 (define-extension generic-fn-symbol fn-extension)

Obeys the same syntax as define, except that the symbol for extension is expected to be bound to a generic function. Only generic functions can be extended.

"Extension" means that the case presented by the given function is examined after all the preceding cases. So if the extension function has the same argument pattern as any of the preceding cases, will not have any effect on the behaviour of the generic function.

You can also use define instead of define-extension.

gfn, gfn:, define-override

1 (define-override generic-fn-symbol fn-override)

Obeys the same syntax as define, except that the symbol for extension is expected to be bound to a generic function. Only generic functions can be overridden.

"Override" means that the case presented by the given function is examined before all the preceding cases of the generic function. This means that if an overriding function has the same argument pattern as any preceding case, it'll completely take over and not give the preceding case a chance.

See also
gfn, gfn:, define-extension

1 (append! list1 list2 ... listN)

Modifies the tail of list1 to point to list2, tail of list2 to list3 etc. All except listN are modified.

For example -

1 (define a (list 1 2 3 4 5))
2 (define b (list 10 20 30 40 50))
3 (append! a b)
4 (print a)

will print

1 (1 2 3 4 5 10 20 30 40 50)
